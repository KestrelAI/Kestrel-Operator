title: "MCP"

The Kestrel MCP server exposes workflow management as tools that AI coding agents (Cursor, Claude Desktop, Windsurf, etc.) can call directly. This lets you create, modify, and monitor workflows through conversational AI in your IDE.

## Setup

### Cursor

Add the Kestrel MCP server to your Cursor configuration at `~/.cursor/mcp.json`:

```json
{
  "mcpServers": {
    "kestrel": {
      "command": "npx",
      "args": ["-y", "@kestrelai/mcp-server"],
      "env": {
        "KESTREL_API_KEY": "your-api-key"
      }
    }
  }
}
```

## Available Tools

Once connected, the following tools are available to the AI agent:

### Workflow Management

| Tool | Description |
|------|-------------|
| `list_workflows` | List all workflows with optional status filter |
| `get_workflow` | Get full details of a specific workflow |
| `create_workflow` | Create a new workflow from a definition |
| `update_workflow` | Update an existing workflow's configuration |
| `delete_workflow` | Delete a workflow |
| `activate_workflow` | Activate a paused workflow |
| `pause_workflow` | Pause an active workflow |
| `duplicate_workflow` | Create a copy of an existing workflow |

### Workflow Generation

| Tool | Description |
|------|-------------|
| `generate_workflow` | Generate a workflow from a natural language description |

### Execution Management

| Tool | Description |
|------|-------------|
| `list_executions` | List workflow executions with filters |
| `get_execution` | Get execution details including step outputs |
| `trigger_workflow` | Manually trigger a workflow with custom data |
| `test_workflow` | Run a workflow test with mock trigger data |

### Catalog

| Tool | Description |
|------|-------------|
| `list_triggers` | List available trigger types and their configurations |
| `list_actions` | List available action types and their configurations |
| `list_integrations` | List connected integrations and their status |

## Example Conversations

**Creating a workflow:**

> "Create a Kestrel workflow that triggers when a PagerDuty alert fires, runs Kubernetes RCA, and posts the results to #incidents on Slack"

The AI agent calls `generate_workflow` with your description, then `create_workflow` to save it, and `activate_workflow` to turn it on.

**Checking execution history:**

> "Show me the last 5 failed workflow executions and what went wrong"

The agent calls `list_executions` with a `failed` status filter, then `get_execution` for each to retrieve step-level error details.

**Modifying an existing workflow:**

> "Update my incident response workflow to also create a Confluence postmortem page after the RCA completes"

The agent calls `list_workflows` to find it, `get_workflow` for the current definition, then `update_workflow` with the added step.

**Investigating observability:**

> "Which of my workflows have failed in the last 24 hours? Show me what went wrong"

The agent calls `list_executions` filtered by time range and failure status, then retrieves details for each.

## Resources

The MCP server also exposes read-only resources that provide context to the AI agent:

| Resource | Description |
|----------|-------------|
| `kestrel://workflows` | List of all workflows with status |
| `kestrel://workflows/{id}` | Full workflow definition |
| `kestrel://catalog/triggers` | Available trigger types |
| `kestrel://catalog/actions` | Available action types |
| `kestrel://integrations` | Connected integration status |

## Authentication

The MCP server authenticates using the `KESTREL_API_KEY` environment variable. Generate an API key from **Settings → API Keys** in the Kestrel dashboard.

<Warning>
The API key has the same permissions as your user account. Use a dedicated service account key for shared MCP configurations.
</Warning>
