Workflows are the core of Kestrel. They chain actions across your integrations — triggered by Kubernetes signals, cloud events, Slack requests, PagerDuty alerts, PostHog events, Vercel deployments, custom webhooks, or manual runs. Build them with natural language, a drag-and-drop canvas, the CLI, SDK, or MCP.

{/* Screenshot placeholder: Workflow canvas with a multi-step incident response workflow */}

## Prerequisites

- A Kestrel account at [platform.usekestrel.ai](https://platform.usekestrel.ai)
- At least one integration connected (see [Integrations](/integrations/kubernetes))

## Create Your First Workflow

<Steps>
  <Step title="Navigate to Workflows">
    From the Kestrel dashboard, go to **Workflows → New Workflow**.

    You can start from a blank canvas or choose a template. Templates cover common use cases like incident response, cloud provisioning, and CI/CD automation.

    {/* Screenshot placeholder: New Workflow page with template gallery */}
  </Step>
  <Step title="Describe your workflow in natural language">
    Type a plain-English description of what you want automated. Kestrel's AI generates the workflow steps for you.

    **Example:**

    > When a pod crashes in the payments namespace, trigger root cause analysis, send a Slack message with the root cause to #incidents, and create a Jira ticket assigned to the payments team.

    The AI assembles the trigger, actions, and connections. Review the generated workflow and adjust as needed.

    <Tip>
    You can also build workflows entirely with the drag-and-drop canvas — add trigger and action nodes, connect them, and configure each step manually.
    </Tip>

    {/* Screenshot placeholder: Natural language input generating a workflow */}
  </Step>
  <Step title="Configure the trigger">
    Every workflow starts with a trigger. Select the event source that kicks off the workflow:

    | Trigger Type | Examples |
    |---|---|
    | **Kubernetes Signal** | Pod crash, deployment failure, OOMKill, node pressure |
    | **Cloud Signal** | AWS resource change, scaling event, cost anomaly |
    | **Slack Request** | Message in a channel, slash command |
    | **PagerDuty** | New incident, acknowledgment, resolution |
    | **PostHog** | Product analytics event threshold |
    | **Vercel** | Deployment success or failure |
    | **Custom Webhook** | Any HTTP POST to your workflow's unique endpoint |
    | **Manual** | On-demand execution from the dashboard or CLI |

    Configure trigger-specific filters to control when the workflow fires — for example, limit a K8s trigger to a specific namespace or label selector.
  </Step>
  <Step title="Configure actions and template variables">
    Add action steps to your workflow. Each action targets an integration and performs a specific operation — sending a Slack message, creating a Jira ticket, running an RCA, applying a Kubernetes manifest, and so on.

    Use the **variable picker** to insert dynamic values from the trigger or from previous action outputs into any field. Variables use the `{{variable_name}}` syntax and are type-checked at build time.

    {/* Screenshot placeholder: Action configuration panel with variable picker open */}

    <Note>
    Actions execute sequentially by default. Use parallel branches on the canvas to run actions concurrently.
    </Note>
  </Step>
  <Step title="Set cooldown and save">
    Configure a **cooldown period** to prevent the same trigger from firing the workflow repeatedly in a short window. For example, set a 5-minute cooldown so a pod crash-looping every 30 seconds doesn't generate dozens of duplicate tickets.

    Click **Save** to activate the workflow.
  </Step>
  <Step title="Test the workflow">
    Use the **Test** button to simulate a trigger event and trace the workflow execution step by step. Each action shows its input, output, and execution time.

    {/* Screenshot placeholder: Workflow test run with step-by-step execution trace */}

    <Tip>
    You can also trigger a test run from the CLI with `kestrel workflow test <workflow-id>`.
    </Tip>
  </Step>
</Steps>

## Example Workflows

| Use Case | Trigger | Actions |
|---|---|---|
| Self-healing K8s incident | Pod CrashLoopBackOff | RCA → Slack alert → Apply fix → Jira ticket |
| Cloud provisioning via Slack | Slack message in #infra-requests | Parse request → Provision AWS resource → Confirm in thread |
| CI/CD automation | GitHub PR merged | ArgoCD sync → Run tests → Notify Slack on failure |
| Cost anomaly response | AWS cost spike detected | Investigate resources → Slack alert to #finops → Linear ticket |
| Product analytics alert | PostHog event threshold | Query metrics → Slack summary → PagerDuty if critical |
